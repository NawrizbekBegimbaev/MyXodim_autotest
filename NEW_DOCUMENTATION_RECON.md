2. Общие концепции

2.3 Системные роли (permission sets)

Роль = исключительно UI/system permission. Определяет только:

Какие разделы меню пользователь видит

Какие кнопки и действия в UI ему доступны

Какие API-endpoints он может вызвать

Роль НЕ определяет:

Участие в документообороте (это делают Позиции или прямое назначение User в шаг маршрута)

Бизнес-полномочия (право подписывать, утверждать — это тоже через Позиции)

Видимость конкретного документа (это определяется авторством или участием в маршруте)

Позиция — место в оргструктуре организации, которое может занимать один или несколько сотрудников. Используется для маршрутов согласования документов.

Правила:

Позиция может иметь нескольких сотрудников (если позиция занята несколькими людьми)

Одному сотруднику можно назначить несколько позиций (совмещение)

Уникальность: комбинация (Сотрудник + Подразделение + Должность) должна быть уникальной — нельзя дважды назначить одного сотрудника на одну и ту же должность в одном подразделении

Позиции создаются и редактируются Администратором организации в Client UI (раздел "Позиции")

Позиции — локальные для организации, не синхронизируются с 1С

2.5 Документ = Контент + Маршрут (независимые объекты)

Ключевой принцип: Шаблон и Маршрут — независимые переиспользуемые объекты. Они не связаны "жёстко"; документ при создании получает каждое отдельно.

Концепты

Шаблон (Template) — переиспользуемая заготовка контента:

Название по умолчанию, структура формы с prefilled значениями, типовое описание, PDF-шаблон (опционально)

Маршрута НЕ содержит

Маршрут (Route) — независимый переиспользуемый объект процесса согласования:

Шаги с targets (User/Позиция), действиями, типом (последовательный/параллельный)

Хранится отдельно, может использоваться с любым документом

Документ = Контент + Маршрут. Контент — что подписывают; Маршрут — кто и в каком порядке.

Пример независимости

Один маршрут "Стандартное подписание директором" может использоваться для шаблонов "Заявление на отпуск", "Командировочное удостоверение", ad-hoc документа.

Один шаблон "Заявление на отпуск" может использоваться с разными маршрутами (для HR — через HR-менеджера; для IT — через тимлида).

Создание документа — 2 независимых выбора

Выбор контента: из шаблона (prefilled + редактирование), или ad-hoc (свободный ввод + файл)

Выбор маршрута: готовый из списка, или ad-hoc (построить на лету)

Любая комбинация допустима (4 варианта): шаблон+готовый маршрут, шаблон+ad-hoc, ad-hoc контент+готовый маршрут, ad-hoc+ad-hoc.

UC-3.1: Создание документа (unified flow)

Пользователь проходит 2 независимых шага: выбор контента и выбор маршрута.

Шаг 1 — Выбор контента:

Вариант A: Из шаблона

Сотрудник открывает Создать заявку → выбирает Из шаблона

Выбирает шаблон (например, "Заявление на отпуск")

Поля формы предзаполняются из шаблона

Пользователь редактирует (ФИО, даты, суммы и т.д.)

Вариант B: С чистого листа (ad-hoc)

Сотрудник открывает Создать заявку → выбирает С чистого листа

Вводит название + описание

Прикрепляет файл (PDF / DOCX / JPG — один или несколько)

Шаг 2 — Выбор маршрута:

Вариант A: Готовый маршрут

Выбирает маршрут из списка ранее созданных (например, "Стандартное согласование директором")

Вариант B: Маршрут на лету

В визуальном конструкторе строит под конкретный документ:

Добавляет шаги (User или Позиция)

Выбирает действие (Согласовать / Утвердить / Подписать)

Настраивает порядок (последовательный / параллельный)

Шаг 3 — Отправка:

Пользователь сохраняет черновик или отправляет

При отправке запускается выбранный маршрут

Пример комбинации (комиссия по инциденту):

Контент: С чистого листа — "Акт о происшествии №123" + PDF
Маршрут: На лету:

Шаг 1 (параллельно): Согласовать → User: Иванов, Петров, Сидоров

Шаг 2: Подписать → Позиция: "Директор"

Другой пример (стандартный отпуск):

Контент: Из шаблона "Заявление на отпуск" — prefilled даты редактируются
Маршрут: Готовый "Отпуск через HR" — выбор из списка

Ограничения ad-hoc контента:

Нет структурированных полей формы (только title + description + attachments)

Нет PDF-генерации (используется приложенный файл)

В 1С отправляется только файл, без данных формы

UC-3.2: Согласование документа

Пользователь открывает Входящие

Видит документ, требующий действия

Открывает документ, просматривает

Выбирает действие: Согласовать / Утвердить / Подписать / Отклонить / Вернуть

Подтверждает

UC-3.3: Параллельное согласование

Система одновременно отправляет задачи всем участникам шага

Каждый независимо принимает решение

Шаг завершается когда все выполнили

Если хоть один отклонил — процесс останавливается

Участник выбирает Вернуть + комментарий

Документ → статус "На доработке"

Автор видит документ, вносит правки

Отправляет повторно

Процесс начинается заново с первого шага

UC-3.5: Управление шаблонами (Администратор / Менеджер)

Открывает Шаблоны → Создать (или импорт из 1С)

Заполняет:

Название шаблона

Структура формы (поля с типами: text, number, date, select, …)

Значения по умолчанию

Шаблон PDF (опционально)

Сохраняет

Правила:

Шаблон не содержит маршрут

Переиспользуется в любом документе (пользователь выбирает маршрут отдельно)

Редактирование/удаление — Админ или Менеджер

UC-3.5b: Управление маршрутами (Администратор / Менеджер)

Маршрут = переиспользуемая последовательность шагов, независимая от шаблонов.

Открывает Маршруты → Создать

Даёт имя маршруту (например, "Стандартное согласование директором")

В визуальном конструкторе настраивает шаги:

Добавляет шаги (drag-and-drop)

Для каждого шага выбирает target:

User — конкретный сотрудник (поиск по телефону/ФИО/ПИНФЛ), ИЛИ

Позиция — слот оргструктуры (поиск из списка позиций)

Выбирает тип действия: Согласовать / Утвердить / Подписать

Выбирает режим шага: последовательный / параллельный

Сохраняет

Правила:

В одном шаге может быть несколько targets (несколько User или несколько Позиций)

Нельзя смешивать типы targets в одном шаге (или только User, или только Позиции)

Маршрут доступен для выбора при создании любого документа

Удаление маршрута — только если он не привязан к активным документам

Заполняет: телефон, имя, фамилия, должность, подразделение

Изменить: должность, подразделение

Назначить роль (только Администратор):

Открывает карточку сотрудника

Меняет роль: Сотрудник → Директор / Менеджер / Администратор

Сохраняет

При следующем входе сотрудника новые права применяются

Правила назначения ролей:

Менеджер и Директор не могут менять роли

Администратор может назначать Директора, Менеджера и другого Администратора

Нельзя снять роль Администратора, если это единственный Администратор в организации (блокировка системой)

Роль ортогональна Позиции: можно иметь роль "Сотрудник" и быть на Позиции "Директор" — это нормально

Отключить: статус → "Отключен", доступ теряется, история сохраняется

UC-3.7: Управление позициями (Администратор организации)

Создать позицию:

Открывает Позиции → Создать

Заполняет:

Название позиции (например, "Менеджер отдела кадров")

Подразделение (из справочника 1С)

Должность (из справочника 1С)

Назначенные сотрудники (один или несколько)

Сохраняет

Редактировать:

Изменить название

Сменить подразделение или должность

Добавить/убрать сотрудников

Удалить:

Если позиция используется в маршрутах → запрет удаления, показать список шаблонов

Если не используется → удалить с подтверждением

Ограничения:

Название позиции должно быть уникальным в рамках организации

Уникальность назначения: комбинация (Сотрудник + Подразделение + Должность) уникальна — один сотрудник не может быть назначен дважды на одну и ту же должность в одном подразделении

Сотрудник может быть назначен на несколько позиций (совмещение — разные Подразделение или Должность)

Нельзя оставить позицию пустой, если она используется в активных маршрутах

1С может инициировать документ и отправить его в HUB для прохождения согласования/подписания.

Auth: Bearer {tenant JWT}

  "externalId": "1C-doc-guid",
  "title": "Приказ о переводе",
  "templateId": "optional-template-id",
  "formData": { ... },
  "route": {
    "steps": [
      {
        "type": "sequential",
        "action": "approve",

          { "kind": "position", "id": "pos-director-uuid" }

      },
      {
        "type": "sequential",
        "action": "sign",

          { "kind": "user", "pinfl": "32607965180052" }

  "metadata": {
    "createdBy": "1C-user-info",
    "originDocumentNumber": "ПР-2026-0042"

6.4 Направление: BusinessHub → 1С (Event Queue API / Pull Model)

Контекст: 1С не умеет принимать push-запросы от внешних систем. Поэтому прямой вызов HUB → 1С невозможен. Вместо этого BusinessHub публикует события в свою очередь, а 1С периодически их забирает (pull).

6.4.1 Концепция
HUB генерирует события → сохраняет в Event Queue (tenant-scoped)
                                    ↓
1С периодически вызывает GET /events?after=lastSeen  (long polling)
                                    ↓
1С обрабатывает события → ACK или Delete (управляет сам)

Это аналог Kafka consumer group по паттерну, но через простой HTTP:

HUB = producer + broker (хранит события)

1С = consumer (полностью управляет чтением, acknowledgement, удалением)

Pull вместо push — 1С держит контроль над ритмом

6.4.2 Tenant-based Row-Level Security (RLS)

Каждая организация (tenant) имеет:

Уникальный tenantId (GUID организации)

API key / JWT для аутентификации 1С-клиента

Все события изолированы: 1С видит только события своего tenant

RLS enforcement:

JWT содержит tenantId claim

Каждый запрос фильтруется на уровне БД по tenant_id = jwt.tenantId

Попытка запросить события другого tenant → 403 Forbidden

6.4.5 API Endpoints

Base URL: /api/v1/integration

Auth: Bearer JWT с tenantId claim или API key

Fetch events (long polling):

GET /events?after={sequenceNumber}&limit=100&waitSeconds=30

after — последний обработанный sequenceNumber (0 для начала)

limit — макс. событий в ответе (default 100, max 500)

waitSeconds — long polling timeout (0 = немедленный ответ, max 60)

Response:

json
{
  "events": [ /* array of events */ ],
  "hasMore": true,
  "nextCursor": 1145
}

Поведение:

Если events есть → возвращает сразу

Если events нет → держит соединение до waitSeconds или до появления event

Если таймаут → пустой массив, 1С повторяет запрос

Acknowledge event (optional):

POST /events/{eventId}/ack

Помечает событие как обработанное. 1С может опционально вызывать для tracking.

Delete event:

DELETE /events/{eventId}

Удаляет событие из очереди навсегда. После этого HUB его не вернёт.

Bulk delete:

POST /events:bulkDelete
Body: { "eventIds": [...] }

Fetch PDF:

GET /documents/{documentId}/pdf

Возвращает PDF файл документа (если уже сгенерирован или ad-hoc прикреплён).

6.4.6 Retention (хранение событий)

HUB не удаляет события автоматически

1С полностью контролирует удаление через DELETE /events/{id} или bulkDelete

По умолчанию событие хранится 30 дней (configurable per tenant) — затем auto-delete с уведомлением в audit log

1С видит все события, даже уже "acknowledged", пока не удалит

6.4.7 Polling Pattern (рекомендация для 1С)
pseudo
cursor = loadLastCursor()  // persisted in 1С
loop {
  response = GET /events?after={cursor}&waitSeconds=30
  for event in response.events {
    process(event)             // 1С обрабатывает
    if processed_ok {
      DELETE /events/{event.eventId}  // убирает из очереди
    }
    cursor = max(cursor, event.sequenceNumber)
  }
  saveLastCursor(cursor)
}
6.4.8 Почему такой подход

Преимущество

	

Объяснение




1С не принимает push

	

Архитектурное ограничение 1С обходится через pull




1С управляет темпом

	

Если 1С занят — просто позже прочитает; HUB не нагружает




Idempotency

	

1С может повторно прочитать и обработать событие при сбое




Изоляция

	

Tenant-based RLS гарантирует изоляцию данных организаций




Простота

	

Только HTTP/JSON — не нужен Kafka, RabbitMQ и т.п.




Отсутствие потерь

	

События хранятся до явного удаления 1С-ом





6.4.9 Ограничения

Задержка = интервал polling (обычно 1–30 секунд, приемлемо для документооборота)

Не для realtime — если нужна реакция < 1 сек, не подходит

Порядок событий — в рамках tenant строго по sequenceNumber; между tenant'ами порядка нет (и не нужен)

7. Электронная подпись — EIMZO
7.1 Обзор

Финальное подписание документа осуществляется через E-IMZO (E-IMZO.uz) — государственную систему электронной цифровой подписи Узбекистана. Без EIMZO-подписи документ не считается юридически значимым.

7.2 Требование

Каждый документ, дошедший до шага "Подписать" (action: sign), должен быть подписан через EIMZO.

Шаги "Согласовать" и "Утвердить" — обычная web-кнопка в UI (без EIMZO)

Шаг "Подписать" — только через EIMZO (ключ, PIN, браузерное расширение)

7.3 Архитектура EIMZO Integration
User browser → EIMZO browser plugin (установлен пользователем)
                          ↓
               Доступ к ключам (usb-token, файловый ключ)
                          ↓
               PIN-код подтверждение
                          ↓
               Подпись PDF (PKCS#7 / .p7s)
                          ↓
HUB Client UI → отправляет подпись на backend
                          ↓
HUB Backend → сохраняет подпись + валидирует через E-IMZO API
                          ↓
Документ → статус "Подписан", шаг пройден
7.4 Требования к UI (Client UI)

E-IMZO JavaScript library интегрируется в Client UI:

Библиотека: официальный e-imzo.js (загружается с e-imzo.uz или копия на backend)

Проверка наличия плагина: EIMZO.checkVersion()

Получение списка сертификатов пользователя: EIMZO.listAllUserKeys()

Подпись: EIMZO.createPkcs7(base64Pdf, certificate, password) → возвращает PKCS#7 signed blob

UI flow на шаге подписания:

Пользователь открывает документ в статусе "Требует подписи"

UI проверяет установку EIMZO plugin

Если не установлен → ссылка на e-imzo.uz/main/downloads с инструкцией

UI запрашивает список ключей → показывает пользователю выбор сертификата

Пользователь выбирает свой ключ (обычно ПИНФЛ совпадает с профилем)

UI вызывает EIMZO.sign() → открывается диалог для ввода PIN

После ввода PIN — подпись сформирована (PKCS#7)

UI отправляет signed blob в backend

Backend валидирует + сохраняет

Валидация на backend:

8. Ключевые бизнес-правила

Один человек = один номер телефона. Может состоять в нескольких организациях

Шаблон и Маршрут — независимы. Шаблон = заготовка контента (без маршрута); Маршрут = независимый объект, переиспользуется в разных документах

Документ = Контент + Маршрут. Контент: из шаблона или ad-hoc; Маршрут: готовый из списка или ad-hoc. Любая комбинация допустима

Строгая очередность. Действовать может только участник текущего шага

Отклонение — конец. Возврат — доработка автором

Обмен с 1С асимметричный: 1С → HUB через push (1С сам отправляет данные справочников), HUB → 1С через Event Queue API (1С pull-ит события). Прямой push от HUB невозможен — 1С не принимает входящие запросы

Изоляция организаций. Одна организация не видит данные другой

Все действия записываются. Полная история изменений (аудит)

Данные не удаляются. Только деактивируются

Системные роли защищены. 4 системные роли (Администратор, Директор, Менеджер, Сотрудник) нельзя удалить или изменить. Default роль нового пользователя — Сотрудник; Администратор вручную назначает Директора, Менеджера или Администратора

Admin UI не настраивает 1С. Интеграция настраивается внутри организации через Client UI

Маршрут адресуется User или Позиции. Роль (системная) не используется в маршрутах — только для доступа в UI

Позиции локальные. Не синхронизируются с 1С, создаются Администратором организации

Документы могут иметь 3 источника: созданы в HUB (по шаблону или ad-hoc) или импортированы из 1С через Integration API

Финальное подписание — только через EIMZO. Шаги "Согласовать"/"Утвердить" через UI-кнопку; шаг "Подписать" требует ключа EIMZO + PIN. Пользователь без ПИНФЛ в профиле подписать не может

Сотрудники филиалов видят документы всей компании

Нет структурированного документооборота между офисами

Workflow не может «уходить» в филиал и возвращаться обратно

Цель

Ввести филиальную структуру с чёткими правилами маршрутизации:

Документ из head → любой branch ✅

Документ из branch → только head ✅

Документ из branch → любой branch
