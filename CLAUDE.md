## 2. Особенности окружения

**OTP:** dev-окружение принимает ЛЮБОЙ 6-значный код. В тестах используем 
константу `TEST_OTP = "123456"` — никакого SMS, никакого backdoor, никаких 
API-вызовов для получения кода.

**EIMZO:** реальная ЭЦП-флешка подключается физически к локальной машине, 
на которой запускаются тесты.
- Локальный прогон: работает (флешка воткнута, плагин видит её)
- CI: НЕ работает (нет физической флешки) — тесты с подписанием помечаем 
  маркером `@pytest.mark.eimzo_local_only`, в CI они пропускаются
- PIN-диалог EIMZO — нативное окно ОС, Playwright им не управляет. 
  Предполагаем что PIN запомнен в плагине. Если нет — пометь TODO в коде 
  и выведи в открытые вопросы
- Запуск тестов с EIMZO — только в headed-режиме (`--headed`), не headless
- Используем системный Chrome (`channel="chrome"`) а не bundled Chromium, 
  потому что плагин установлен в системный браузер

**Cleanup:** по BRD данные не удаляются. Тестовые организации создаём с 
префиксом `[E2E] ` в имени и UUID-суффиксом. Maintenance-тест чистит их 
раз в неделю через Admin UI.

## 3. Главный сценарий (cross-app, multi-user)

ОДИН большой E2E тест в `tests/e2e/test_admin_to_client_full_flow.py`.

**Терминология (важно):** в реальном UI домен «Позиция» отсутствует — 
везде называется «Должность». POM-имена сохраняем доменные (BRD), 
но i18n-ключи и локаторы используют слово «Должность».

**Флоу через Mock 1C — это и есть киллер-кейс системы:**
данные сотрудников/должностей рождаются в Mock 1C и push-кнопкой 
«Отправить» уходят в Client UI. Локальное создание в Client UI тоже 
работает, но тестируется отдельно в `tests/client/`.

```
1.  Admin UI: Super Admin логинится
2.  Admin UI: создаёт компанию + первого Администратора
3.  Admin UI: проверяет компанию в списке со статусом "Активна"

4.  Client UI (контекст автора=админ из шага 2): логин по OTP →
    автоактивация → /integration (или /organization) → копируем
    "Ключ интеграции 1С"

5.  Mock 1C: вставляет ключ → подключение успешно
6.  Mock 1C: создаёт должность (любое имя, например "Согласующий-{uuid}")
    → жмёт "Отправить" на странице Должности
7.  Mock 1C: создаёт сотрудника #2 (телефон A из phone pool)
    → жмёт "Отправить" на странице Сотрудники
8.  Mock 1C: создаёт сотрудника #3 (телефон B из phone pool),
    привязанного к должности из шага 6 → жмёт "Отправить"

9.  Client UI (автор): /members → видит сотрудников #2, #3
    /positions → видит должность из шага 6
    /org-positions → видит штатную позицию (создаётся только из 1С)

10. Client UI: создаёт Маршрут /routes
    - шаг 1: approve, target = сотрудник #2
    - шаг 2: sign,    target = должность из шага 6 (резолвится в #3)
11. Client UI: создаёт Шаблон /templates
12. Client UI: создаёт документ /documents из шаблона + маршрут → отправляет
13. Client UI: видит документ в "Мои документы" → "В работе"

14. Client UI (контекст согласующего #2): логин по тел A →
    автоактивация → /inbox → согласовывает

15. Client UI (контекст подписанта #3): логин по тел B →
    автоактивация → /inbox → подписывает через EIMZO
    @pytest.mark.eimzo_local_only

16. Client UI (контекст автора): page.reload() → документ "Завершён"
17. Mock 1C: специалист видит документ во "Входящие документы"
```

**Терминология Client UI (по факту разведки):**
- BRD «Сотрудник»  → UI «Пользователь» (раздел /members, кнопка "Добавить сотрудника")
- BRD «Позиция»    → UI «Штатная позиция» (раздел /org-positions),
                     создаётся ТОЛЬКО через 1С — alert на странице
- UI «Должность»   → отдельный реестр наименований (/positions),
                     создаётся локально или через 1С
- BRD «Branch»     → UI «Филиал» (/branches), tree-структура
- Категории        → /categories (древовидные)
- Роли             → /roles (кастомные кроме 4 системных из BRD §3.5)
- Документы:
    /inbox        — "Требуют подписи" (Входящие на действия)
    /documents    — "Документы" (всё что доступно пользователю + автор)

ОДИН тест-метод, разбит на @allure.step. НЕ дробить — ценность именно 
в непрерывности cross-app сценария.

**Push-кнопка "Отправить" в Mock 1C** — отдельная для каждой сущности 
(должность отправляется со страницы Должности, сотрудник — со страницы 
Сотрудники). Не одна общая.

**Альтернативная версия** `test_admin_to_client_full_flow_no_eimzo` — 
то же самое, но останавливается на шаге 15 (после согласования, без 
подписания). Эта версия гоняется в CI.

Маркер `@pytest.mark.serial` для главного E2E — не запускать параллельно 
(одна флешка на машине).

## 3a. Категории тестов — что писать и в каком объёме

Тесты делятся на категории. Каждая категория — отдельная папка с понятным 
именем. Маркеры pytest позволяют запускать категории отдельно.

### Smoke (positive happy-path, базовая работоспособность)

**Папка:** `tests/smoke/`  
**Маркер:** `@pytest.mark.smoke`  
**Время:** до 10 минут на всю папку  
**Запускать:** на каждый PR

Только самое критичное — если упадёт, релизить нельзя:
- Все три UI открываются
- Super Admin логинится в Admin UI
- Сотрудник логинится в Client UI через OTP
- Создание организации проходит до конца
- Создание документа из шаблона проходит до отправки

### Positive (основные пользовательские сценарии)

**Папка:** `tests/client/`, `tests/admin/`, `tests/mock1c/`  
**Маркер:** `@pytest.mark.positive`

Покрытие всех ключевых юзкейсов из BRD §3.3 и §4.3:
- UC-3.1 Создание документа: 4 комбинации (шаблон+готовый маршрут, 
  шаблон+ad-hoc, ad-hoc контент+готовый маршрут, ad-hoc+ad-hoc)
- UC-3.2 Согласование документа
- UC-3.3 Параллельное согласование (все участники одобряют)
- UC-3.4 Возврат на доработку и повторная отправка
- UC-3.5 Управление шаблонами (CRUD)
- UC-3.5b Управление маршрутами (CRUD)
- UC-3.6 Управление сотрудниками (добавить, изменить, отключить)
- UC-3.7 Управление позициями (CRUD)
- UC-4.1 Создание организации
- UC-4.2 Просмотр и управление организацией
- UC-4.3 Включение/отключение организации

### Negative (ошибки, валидация, неверные данные)

**Папка:** рядом с positive, файл `test_*_negative.py`  
**Маркер:** `@pytest.mark.negative`

Каждый юзкейс должен иметь негативный двойник. Минимум:

**Логин (Client UI):**
- Неверный OTP-код (не 6 цифр) → ошибка валидации
- Пустой телефон → кнопка "Отправить" неактивна
- Невалидный формат телефона → ошибка
- Телефон не существует в системе → "Доступ запрещён"
- Отключённый пользователь → "Доступ запрещён"

**Логин (Admin UI):**
- Неверный пароль → ошибка
- Пустые поля → кнопка неактивна

**Создание организации:**
- ИНН уже существует → ошибка дубликата (по BRD §4.5)
- Невалидный ИНН (не 9 цифр, буквы) → ошибка валидации
- Пустое название → ошибка
- Пустой телефон администратора → ошибка
- Невалидный формат телефона администратора → ошибка
- Дубликат телефона администратора → ошибка (BRD §4.5)

**Создание сотрудника:**
- Дубликат телефона в той же организации → ошибка
- Невалидный формат телефона → ошибка
- Пустые обязательные поля → ошибка

**Создание Позиции:**
- Дубликат комбинации (Сотрудник + Подразделение + Должность) → ошибка 
  (BRD §3.7 "Уникальность назначения")
- Дубликат имени Позиции → ошибка
- Пустое имя → ошибка

**Удаление Позиции:**
- Позиция используется в активном маршруте → запрет удаления, 
  показан список шаблонов (BRD §3.7)

**Создание маршрута:**
- Шаг без targets → ошибка
- Смешанные типы targets в одном шаге (User + Position) → ошибка (BRD §3.5b)

**Создание документа:**
- Отправка без выбора маршрута → ошибка
- Отправка без обязательных полей формы → ошибка

**Действия в маршруте:**
- Попытка действовать на чужом шаге (не свой turn) → действие недоступно
- Отклонение без комментария → ошибка (BRD §2.7 "комментарий обязателен")
- Возврат без комментария → ошибка

### Edge cases (граничные случаи из BRD §9)

**Папка:** `tests/edge_cases/`  
**Маркер:** `@pytest.mark.edge_case`

Все 10 кейсов из таблицы §9 BRD:
- Позиция без сотрудников → задача падает на Администратора
- User в шаге отключён → задача падает на Администратора
- Сотрудник уволен во время согласования (единственный на позиции) → 
  передача Администратору
- Сотрудник уволен во время согласования (несколько на позиции) → 
  остальным участникам
- Попытка удалить Позицию которая в активном маршруте → запрет
- Параллельный шаг: 1 отклонил → весь процесс останавливается
- Возврат на доработку → процесс начинается заново с первого шага
- Пользователь в 2+ организациях → экран выбора после OTP
- Ошибка 1С при отправке → статус "Ошибка экспорта"

### RBAC (права доступа)

**Папка:** `tests/auth/test_rbac.py`  
**Маркер:** `@pytest.mark.rbac`

Параметризованные тесты по таблице §3.5 BRD. Для каждой из 4 ролей 
(Администратор, Директор, Менеджер, Сотрудник):
- Видимые пункты меню соответствуют таблице
- Невидимые пункты действительно не отображаются
- Прямой переход по URL раздела куда нет доступа → 403 или редирект

Аналогично для Admin UI ролей (Super Admin, Admin) по таблице §4.4.

### E2E (главный сквозной сценарий)

**Папка:** `tests/e2e/`  
**Маркер:** `@pytest.mark.e2e` + `@pytest.mark.serial`

Описан в §3. Не дробить.

### Maintenance (обслуживание тестового окружения)

**Папка:** `tests/maintenance/`  
**Маркер:** `@pytest.mark.maintenance`  
**Запускать:** вручную или по расписанию (раз в неделю)

- Чистка тестовых организаций с префиксом `[E2E]` старше 7 дней
- Проверка что пул тестовых телефонов цел и не "залип"

## 4. Multi-user через несколько BrowserContext

КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО переключать пользователей через UI logout → login 
в середине сценария.

Правила:
- Каждый участник = отдельный BrowserContext (изолированные cookies, localStorage)
- Все контексты создаются в начале теста, живут до конца, закрываются в teardown
- Между действиями разных юзеров — page.reload() для подтягивания свежих данных
- НЕ использовать одну page для нескольких юзеров

Для главного теста: 4 контекста — admin (Admin UI), author, approver, signer 
(все три на Client UI).

ВАЖНО: в главном E2E пользователи создаются ВНУТРИ теста, поэтому 
storage_state использовать нельзя — логинимся через UI прямо в тесте.

В обычных тестах (не главный E2E) — Super Admin логинится в session-фикстуре, 
storage_state сохраняется в `.auth/super_admin.json` и переиспользуется во 
всех тестах сессии.

UI-флоу logout/login тестируется ОТДЕЛЬНО в `tests/auth/test_logout.py` — 
там это и есть предмет проверки.

## 5. Test data

Каждый тест работает в СВОЕЙ организации, созданной через UI в фикстуре:
- Имя: `[E2E] {timestamp} {uuid4().hex[:6]}` (видно в админке что тестовое)
- ИНН: генерируем валидный 9-значный UZ ИНН с чек-суммой
- ID для 1С: uuid4
- Телефон Администратора: из пула TEST_PHONE_POOL

Пул телефонов:
- Минимум 50 номеров `+998900000001..050` для параллельных прогонов
- Атомарная выдача через `data/phone_pool.py` с filelock (для xdist)
- После теста телефоны возвращаются в пул

Cleanup:
- В teardown — НЕ пытаемся удалить через UI (медленно)
- Префикс `[E2E]` в имени → maintenance-тест раз в неделю чистит всё что 
  старше 7 дней через Admin UI

## 8. Page Object — правила

1. Класс на каждую страницу. В `__init__` принимает `page: Page`
2. Локаторы — приватные атрибуты, инициализируются в `__init__`
3. Методы — действия и геттеры. БЕЗ ассертов. Возвращают `Self` или следующую Page
4. BasePage содержит `goto`, `wait_loaded`
5. Type hints обязательны во всех методах

Приоритет локаторов:
1. `get_by_role(role, name=...)` — кнопки, ссылки, инпуты с label
2. `get_by_label(...)` / `get_by_placeholder(...)` — формы
3. `get_by_test_id(...)` — если фронт проставил data-testid
4. `get_by_text(...)` — для уникальных текстов
5. CSS — только если выше не работает
6. Классы Svelte (`.svelte-jspc58`, `.svelte-iko9rr`) — ЗАПРЕЩЕНЫ, 
   меняются между билдами
7. XPath — запрещён без обоснования в комментарии

## 9. Стабильность и скорость

ЗАПРЕЩЕНО:
- `time.sleep()`, `page.wait_for_timeout()` для синхронизации
- `page.wait_for_selector()` без `state=...`
- `if locator.is_visible(): ...` для управления потоком — это race condition
- `try/except Exception` для "стабильности"
- Хардкод URL, телефонов, таймаутов в коде тестов
- Ассерты внутри Page Object
- Глобальное состояние между тестами
- Зависимости между тестами

ИСПОЛЬЗУЕМ:
- `expect(...).to_*` — auto-retrying assertions
- `page.expect_response(lambda r: "/api/v1/documents" in r.url)` — для 
  ожидания завершения запроса бэкенда после клика, БЕЗ парсинга ответа
- `page.wait_for_load_state("networkidle")` — для тяжёлых SPA-страниц
- Базовые таймауты 10s expect / 15s navigation — в pytest.ini, не в коде

## 10. Storage state — для тестов кроме главного E2E

Session-scope фикстуры выполняют UI-логин ОДИН раз и сохраняют состояние:

```python
@pytest.fixture(scope="session")
def super_admin_state(browser, settings):
    ctx = browser.new_context()
    page = ctx.new_page()
    AdminLoginPage(page).login(settings.super_admin_phone, 
                                settings.super_admin_password)
    state_path = ".auth/super_admin.json"
    ctx.storage_state(path=state_path)
    ctx.close()
    return state_path
```

В тесте: `browser.new_context(storage_state=super_admin_state)` — открывается 
уже залогиненный.

Это работает для постоянных пользователей (Super Admin). Для тестовых 
сотрудников, которые создаются в самом тесте — логинимся через UI прямо 
в тесте.
