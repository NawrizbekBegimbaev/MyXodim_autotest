BUG-001: Фронт молча игнорирует ВСЕ 4xx-ответы от бэка при создании компании
         (нет error toast, нет inline-ошибки)
================================================================================

Severity:           Major
Priority:           P1
Компонент:          Frontend (страница /tenants/new, обработка POST-ответа)

[STILL BROKEN 2026-05-19 verified via MCP] Класс silent-failure
подтверждён в Client UI /members → Add Employee: backend вернул 400
CONSTRAINT_VIOLATION с violations[{"propertyPath":"roleId","detail":"не
должно быть null"}], frontend silent — dialog не закрылся, ни toast, ни
inline-error, ни aria-invalid. То же поведение что Admin UI Tenant Create
изначально.

Описание:
    При создании компании на любой 4xx-ответ бэка фронт НЕ показывает
    пользователю никакого сообщения:
        - нет toast в Notifications-region
        - нет inline-ошибки рядом с полем
        - форма остаётся открытой с введёнными данными

    Пользователь видит что "Создать" не сработала, но не понимает
    причину.

Затронутые сценарии (подтверждены recon'ом 2026-04-28):
    409 INN_EXISTS         — дубль ИНН
    409 SLUG_EXISTS        — дубль slug
    409 PHONE_ADMIN_EXISTS — дубль телефона администратора
    409 PINFL_EXISTS       — дубль ПИНФЛ
    400 constraint-violations — невалидный формат (например буквы в ИНН)

    Все эти ответы содержат человекочитаемое поле "detail" и "title" —
    но фронт его не рендерит.

Шаги воспроизведения (пример для дубль-ИНН):
    1. Создать компанию A с известным ИНН (валидно).
    2. Открыть /tenants/new и попытаться создать B с тем же ИНН.
    3. Нажать "Создать".

Ожидаемый результат:
    Toast/inline с текстом из ответа:
        "Организация с этим ИНН уже зарегистрирована" (для 409 INN_EXISTS)
        "Такой slug уже существует" (для 409 SLUG_EXISTS)
        "Этот телефон используется другим администратором" (для 409 PHONE_ADMIN_EXISTS)
        "Указанный ПИНФЛ уже зарегистрирован" (для 409 PINFL_EXISTS)

Фактический результат:
    region "Notifications alt+T" пуст; форма открыта; пользователь
    в недоумении.

Тело ответа от бэка содержит всё нужное:
    {
      "detail":"Организация с этим ИНН уже зарегистрирована",
      "title":"ИНН уже существует",
      "code":"INN_EXISTS",
      "status":409,
      ...
    }

Доказательство:
    - recon/neg_01_dup_inn.yaml      — пустая Notifications при 409 INN
    - recon/neg_02_dup_slug.yaml     — пустая при 409 SLUG
    - recon/neg_03_dup_phone.yaml    — пустая при 409 PHONE_ADMIN
    - recon/neg_05_inn_letters.yaml  — пустая при 400 constraint-violations
    - recon/neg_api.txt              — все 4xx ответы

Рекомендация:
    Привести обработку ошибок к единой стратегии:
        - на любой 4xx/5xx читать поле detail из ответа (RFC 7807 problem+json)
        - показывать toast "<title>: <detail>"
        - для constraint-violations (BUG-002) дополнительно парсить
          violations[] и подсвечивать конкретные поля формы
        - оставлять форму открытой с введёнными данными

BUG-002: RBAC обходится прямым URL — non-admin роли получают доступ к
         /integration (ключ 1С), /members, /roles, /organization и др.
================================================================================

Severity:           Critical (security — leak ключа интеграции 1С)
Priority:           P0
Компонент:          Frontend Client UI (route guards) + Backend API
                    (отсутствие проверки прав на эндпоинтах)

[PARTIAL FIXED 2026-05-19 verified via MCP, login as EMPLOYEE
+998906145041 tenant [E2E recon] 8dgk1l]
Frontend route-guard добавлен на ВСЕ ключевые admin-URL: /integration,
/roles, /members, /organization, /positions, /departments, /org-positions,
/categories, /persons, /templates, /routes — все редиректят Employee'а
на /forbidden?from=<path>. Исключение: /branches всё ещё рендерится
("Не удалось загрузить филиалы") — единственная дыра в route-map (то же
что NEW FINDING 2026-05-07 в BUG-009). 1C-ключ leak закрыт: UI redirect
работает, /api/v1/integration/1c/key → 401, /api/v1/integration/1c → 500
(не data leak, internal crash).
Backend RBAC HOLE остаётся: GET /api/v1/users/members → 200 (Employee
видит phones/roles/names всех 19 членов), GET /api/v1/users/roles → 200
(Employee видит permission matrix всех ролей включая ADMINISTRATOR).
Эти 2 endpoint'а должны быть admin-only.

Описание:
    Sidebar (меню) Client UI корректно скрывает админ-только разделы
    для не-Administrator ролей. Например, под кастомной ролью FINANSIST
    в меню видны только: «Требуют подписи», «Мои документы»,
    «Мои заявки», «Шаблоны», «Маршруты».

    Однако при ПРЯМОМ переходе по URL пользователь без прав попадает
    на любую административную страницу с полным её содержимым:

        /members      — список Пользователей орг
        /roles        — Роли и права
        /positions    — Должности
        /categories   — Категории
        /routes       — Шаблоны маршрутов
        /organization — Настройки организации
        /integration  — Интеграция с 1С + КЛЮЧ ИНТЕГРАЦИИ ВИДЕН  ⚠️

    Заблокирован прямым URL только /branches (h4 не рендерится).

    КРИТИЧНО: /integration содержит ключ `bh_live_<32 hex>` который
    позволяет push'ить данные сотрудников/должностей/документов в
    компанию через Mock 1C API. Любой залогиненный юзер компании,
    независимо от роли, может этот ключ скопировать.

Предусловия:
    - Орг с кастомной не-admin ролью (например QaTeam, роль FINANSIST)
    - Сотрудник этой орг с не-admin ролью

Шаги воспроизведения:
    1. Логин Администратором, /members → создать сотрудника с ролью
       FINANSIST (или любая не-Administrator кастомная роль).
    2. Logout, залогиниться созданным сотрудником.
    3. Sidebar — админ-разделов нет (RBAC на меню работает).
    4. В адресной строке ввести напрямую:
           https://dev-hub-client.greatmall.uz/integration
    5. Страница загружается, виден ключ интеграции с 1С.
    6. Аналогично открываются /members /roles /organization etc.

Ожидаемый результат:
    Прямой переход на админ-only URL под не-Administrator юзером:
    - либо редирект на /dashboard (или 404)
    - либо страница 403 «Доступ запрещён»
    - бэк должен возвращать 403 на API-эндпоинтах раздела

Фактический результат:
    Страница рендерится полностью с данными. Ключ интеграции 1С
    открыт всем юзерам организации.

Доказательство:
    - scripts/recon_rbac_finansist.py — повторяемый скрипт
    - recon/finrole_04_dashboard_as_finansist.yaml
    - tests/client/test_rbac.py::test_finansist_cannot_access_integration
      (xfail — будет проходить после фикса)

Рекомендация:
    1. Frontend: route guard на каждой админ-странице — проверять
       permission текущего юзера, при отсутствии — redirect /dashboard
       или 403-page.
    2. Backend: ВСЕ админ-эндпоинты должны проверять права на сервере.
       Только защита на фронте недостаточна — POST/GET напрямую через
       devtools обойдут любой UI guard.
    3. Спец-кейс /integration: ключ интеграции должен быть доступен
       только Администратору орг. Замаскировать частично (••••••••<last4>)
       пока юзер явно не нажмёт "Показать", аудит лога просмотров.
    4. Аудит существующих юзеров: проверить логи — кто из не-admin
       юзеров обращался к /api/v1/integrations/* за последние месяцы.

Severity:           Critical (workflow integrity violation)
Priority:           P0
Компонент:          Backend (workflow engine — parallel step semantics)
Найдено:            2026-05-04 ручное QA

[BLOCKED 2026-05-19] Repro требует parallel-step workflow с 2+
активными actor-users + admin отправляет doc. На recon-tenant 0
routes, 4 EMPLOYEE active (можно использовать как параллельных

Описание:
    По BRD §3.4 / §9 параллельный шаг с N участниками: все N должны
    согласовать → шаг complete. Один отклонил → шаг rejected
    (entire workflow stops).

    Фактическое поведение: один из участников нажал "Согласовать" →
    система помечает шаг complete и аналогично "согласует" за
    остальных N-1 участников (они даже не успели открыть документ).

    Это эквивалентно single-actor шагу — параллельность сломана.

        дойдёт до проверки если кто-то торопится и согласует первым.
      - SoD (separation of duties) нарушено.

Ожидаемый результат:
    Параллельный шаг с участниками A, B, C:
      - A approves → шаг status="awaiting" (1/3 completed),
        B and C still pending
      - B approves → шаг status="awaiting" (2/3)
      - C approves → шаг status="completed" → next step starts
      - На любом этапе если X rejects → workflow stops (BRD §9
        правило "1 отклонил → весь process останавливается")

    (или их task просто исчезает без действия).

Рекомендация:
    Backend WorkflowEngine.handleApproval:
      - Если step.type == PARALLEL:
        участник X отметился approved → step.completedActors.add(X)
        if step.completedActors.size < step.totalActors → step
          status remains 'in_progress'
        else → step.status = 'completed', advance to next step
      - Если step.type == PARALLEL и X отметился rejected → step.
        status = 'rejected', workflow.status = 'stopped'

    Регрессионный test:
    test_parallel_step_one_approve_does_not_complete_others.
    Создать parallel step из 2 actors → A approves → проверить
    что B's inbox всё ещё содержит документ как pending.

BUG-034: /env.js dev-stub всё ещё деплоится на оба UI (BUG-027 partial-fix
         tail — Cache-Control починили, но stub не убрали)
================================================================================

Severity:           Minor (BUG-027 был помечен FIXED, но половина проблемы
                    осталась)
Priority:           P3
Компонент:          Frontend deployment (build pipeline / nginx config)
Найдено:            2026-05-19 при MCP sweep — BUG-027 re-verification

Описание:
    BUG-027 [FIXED 2026-05-19] (cache headers) — корректно добавлен
    `Cache-Control: no-store` на `/runtime/env.js`. Но dev-стенд
    одновременно деплоит **двe конкурирующие env-конфигурации**:

        /env.js          → window.__ENV__ = { API_URL: "" }
        /runtime/env.js  → window.__ENV__ = { API_URL: "https://dev-hub-api..." }

    HTML index.html подключает оба:
        <script src="/env.js"></script>
        <script src="/runtime/env.js"></script>

    Сейчас работает только потому что runtime/env.js загружается
    вторым и перезаписывает первое. Но:
      - Если порядок script-tag'ов случайно сменится — broken state.
      - Если /runtime/env.js закэшируется в браузере (Cache-Control
        пока что no-store, но это можно регрессировать) — снова
        broken state.
      - В DevTools видно "лишний" 200 с подозрительным `API_URL:""`,
        что может смутить debugging.

    BUG-027 формально FIXED (login работает, cache headers правильные),
    но root cause (dev-stub env.js) остался. Новый bug под минорный
    cleanup.

Шаги воспроизведения:
    1. Открыть https://dev-hub-client.greatmall.uz/
    2. DevTools → Network → фильтр env.js
    3. Видны два запроса:
       - GET /env.js → 200, body содержит `window.__ENV__ = {API_URL: ""}`
       - GET /runtime/env.js → 200, body содержит правильный API_URL.

Ожидаемое:
    На dev-стенде должен быть только один источник env-конфига
    (либо `/env.js`, либо `/runtime/env.js`). Stub удалить.

Фактическое:
    Два конкурирующих файла. Работает по счастливой случайности
    порядка загрузки.

Рекомендация:
    1. Frontend build: исключить /env.js stub из dev/prod-сборки
       (оставить только для local-dev режима где runtime/env.js
       не генерируется).
    2. nginx config: 410 GONE для /env.js если runtime/env.js
       присутствует.
    3. Build pipeline: fail-build if API_URL пустой в /env.js при
       deploy на не-local target.

Связь:
    - BUG-027 (FIXED but tail issue) — это complement bug.
