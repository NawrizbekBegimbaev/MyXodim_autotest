DOC-001: Конфликт ролей — 5 уровней доступа vs 4 системные роли
================================================================================

Severity:           Critical (блокирует RBAC design / test-suite планирование)
Priority:           P0
Тип:                Documentation conflict
Источник:           BRD 1.0 §2.3 vs BRD 3.0 / Роль - Админ / Пользователи / Физлицо
Найдено:            2026-05-25 при чтении новой документации

Описание:
    BRD 1.0 §2.3 определяет 4 системные роли:
        - Администратор
        - Директор
        - Менеджер
        - Сотрудник
    Эти роли используются в Membership и привязаны к permission matrix.

    BRD 3.0 / Роль - Админ / Пользователи / Физическое лицо определяет
    "Уровень доступа" физлица — 5 значений:
        - Сотрудник
        - HR
        - Руководитель HR
        - Администратор
        - Генеральный директор

    Mapping между двумя моделями НЕ определён:
        - Где "Менеджер" из BRD 1.0?
        - Где "Директор" из BRD 1.0?
        - Является ли "Генеральный директор" BRD 3.0 синонимом "Директор"
          BRD 1.0, или это новая роль?
        - "HR" и "Руководитель HR" — новые роли, не покрытые BRD 1.0.

Импакт на тесты:
    - Невозможно написать параметризованный RBAC suite без чёткого
      mapping'а ролей.
    - Permission matrix (BRD 1.0 §3.5) даёт правила для 4 ролей —
      непонятно как применять к новым 5 уровням.
    - HR-кабинет (расчётный лист/график/отпуск) описан в BRD 3.0 для
      роли "HR" — но эта роль не существует в BRD 1.0 permission
      model.

Вопросы PO:
    1. Какой набор ролей — single source of truth?
    2. Если 5 уровней — это новая модель, как мигрируют существующие
       Membership записи?
    3. HR/Руководитель HR — это отдельные системные роли или просто
       custom-роли поверх 4 базовых?

Действие до ответа:
    Тесты пишутся по BRD 1.0 (4 роли) как baseline, новые роли
    BRD 3.0 помечаются TODO/needs_clarification.

      sub-branches)
    - Сложные org-структуры (head → regional office → branch) не
      имплементируемы

Связь:

BUG-003: GET /documents/new рендерит Spring-style error "Invalid UUID
         string: new" вместо wizard или redirect
================================================================================

Severity:           Major (раздел Documents частично недоступен через URL)
Priority:           P1
Компонент:          Backend (router/controller) + Frontend (no fallback page)
Найдено:            2026-05-25 при попытке проверить UI для BHUB-41/42

Описание:
    Прямой URL https://dev-hub-client.greatmall.uz/documents/new
    рендерит raw Spring exception:

        java.lang.IllegalArgumentException: Invalid UUID string: new

    Backend пытается распарсить путь `/documents/{id}` где `id=new`,
    парсер UUID падает, exception эскалирует до user-facing layer.

Проблемы:
    1. Backend leaks stack trace / class name — security/cosmetic.
    2. Нет распознавания `/documents/new` как wizard-route.
    3. Frontend не имеет catch-all error boundary для UUID parse failure.

Шаги воспроизведения:
    1. Login Client UI.
    2. В адресную строку: /documents/new
    3. Видна raw Java exception страница.

Ожидаемое:
    Либо /documents/new открывает wizard создания документа,
    либо frontend редиректит на /documents с toast "Документ
    не найден".

Фактическое:
    Spring/JVM error page leaks.

Импакт:
    - Любая ссылка извне приложения (email/Slack) с `/documents/new`
      ломает UX.
    - Test'ы которые ожидают wizard по /documents/new — невозможны
      (нужен другой entry point).

Подтверждено через MCP (2026-05-25).

Класс: error-handling / observability (то же что old BUG-028 RFC
7807 type-link, но более серьёзно — здесь raw stack trace).

BUG-006: Route Builder требует "Группы видов документов" вместо
         "Видов документов" — расхождение со спекой BRD 3.0 §27
================================================================================

Severity:           Major (spec drift / unclear product decision)
Priority:           P1
Тип:                Spec drift или intentional simplification — требует
                    подтверждения PO
Компонент:          Frontend Client UI (RouteEditor) + Backend (route schema)

    BRD 3.0 §27 (Маршруты согласований) явно говорит:

    Реальная реализация UI /routes/new:

       охватывает все Виды внутри. Тогда нужно явно описать это
       в BRD и FAQ для пользователей.

Импакт:
    - Тесты `test_route_create_with_doc_type_succeeds` написаны под
      привязку к Виду — нужно переписать под Группу (или skip до
      ясности).

BUG-022: Backend 404-ы засоряют console errors — strict console-guard
         ломается на HR-страницах
================================================================================

Severity:           Minor (DX / test infra)
Priority:           P3
Компонент:          Backend + Frontend (console error noise)
Найдено:            2026-05-26 при recon HR-кабинета

Описание:
    Backend HR endpoints отдают 404 с domain code (например
    `HR_VACATION_BALANCE_NOT_FOUND`) — это легитимный contractual
    empty (вместо 200 с null).

    Frontend ловит 404 и логирует console.error:
        "Failed to load resource: the server responded with a status of 404"

    Наш `_strict_console` autouse fixture в `conftest.py` (см.
    `_CONSOLE_IGNORE` re patterns) уже ignor'ит 4xx как валидное
    negative-поведение. НО — на HR-страницах число таких errors
    становится 5+ за один page load, что засоряет allure-отчёты
    и обходит цель strict guard.

Импакт:
    - DX: test logs шумные, маскируют реальные баги
    - Allure attachments дублируются
    - Невозможно различить "HR пустой по дизайну" vs "HR endpoint
      сломан" по console output

Ожидаемое:
    Либо backend перейти на 200 с empty payload для contractual
    empty (вместо 404), либо frontend gracefully fetch без
    console.error для domain 404.

Подтверждено через MCP (2026-05-26).

Связь:
    - BUG-018 (empty-state copy)
    - BUG-019/020 (general HR backend observability)
